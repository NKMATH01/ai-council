## 6. Lessons Learned & Retrospective

### 6.1 What Went Well (Keep)

- **보안 + 운영 동시 처리**: Phase 1-2를 패러럴로 진행하면서 보안 경계와 운영 가드레일을 동시에 강화. GPT 크로스 리뷰가 hidden issue 발견 효율적.

- **Type-safe 환경변수 전략**: Zod 대신 단순 함수(requireEnv)를 선택. 테스트 환경에서 graceful fallback 가능하면서도 프로덕션 안정성 확보.

- **Clarification Phase 구조화**: 추상적인 "사용자 이해 단계"를 vision/features/technical/resolution 4 phase로 구체화. UI/UX, 프롬프트, 데이터 구조가 모두 정렬됨.

- **Workflow 분리 성공**: useDebate 1,667줄을 3개 workflow + reducer로 분리. 각 workflow의 책임이 명확하고 테스트 가능성 향상.

- **CI/CD 조기 도입**: 보안 blocker 6개를 harness-ops-checklist.md로 명시. 배포 프로세스에 자동 검증 로직 추가.

### 6.2 What Needs Improvement (Problem)

- **Dead code 발생**: debate-effects.ts가 well-designed 하나 integration fail. 애초 설계 시 사용 지점을 확정하지 않았음.

- **유지보수성 목표 미달**: useDebate.ts 200줄 목표 설정이 과도했음. retryFromError(110줄), fetchStream(44줄)이 여전히 인라인.

- **문서 보다 코드 우선**: .env.example, ClarifyRequest.phase 같은 부가 작업이 누락. 보안/운영/기능에 집중하느라 개발자 경험(DX) 문서 미흡.

- **Gap 분류 미숙**: Gap 5건 중 3건이 낮음 심각도인데도 PASS 조건에 영향. Gap 정의 기준이 명확하지 않았음.

### 6.3 What to Try Next (Try)

- **Dead code 정책 수립**: PR 승인 전 "사용되지 않는 코드 = 제거" 규칙 도입. 또는 debate-effects.ts를 useDebate의 일부로 명시적 import.

- **점진적 리팩토링 사이클**: 402줄 → 200줄을 한 번에 하지 말고, Sprint별로 50줄씩 감량 목표. 각 iteration에서 충분한 테스트.

- **DX 문서 템플릿**: 매 Phase 마다 "개발자 가이드" 섹션 추가. .env.example, API 예시, 환경 설정 가이드를 필수 deliverable로.

- **Gap 정의 재정의**: Critical (차단) / Major (기능 영향) / Minor (스타일/문서) 3단계로 분류. PASS 조건은 Critical만 0건.

- **크로스 리뷰 높이기**: GPT와의 리뷰를 "발견만" 하는 수준에서 "지원"까지. 예: "이 부분을 refactor 하면 어떨까?" 제안까지.
